# OneShot (원샷) — 파티 게임 허브 설계서

> 작성일: 2026-06-25
> 상태: 설계 확정 (구현 계획 수립 전)

---

## 0. 한 줄 정의

**OneShot(원샷)** 은 친구끼리 한 **방**에 모여 여러 술/파티 게임을 갈아끼우며 노는 캐주얼 파티 게임 허브다. 첫 게임은 달무티 기반 카드 게임 **Upstage**.

---

## 1. 정체성 & 목표

| 항목 | 결정 |
|---|---|
| 정체성 | 친구끼리 노는 캐주얼 파티 도구 (수익화는 부차) |
| 1순위 성공 기준 | 실제 모임에서 마찰 없이 잘 굴러가는가 |
| 플레이 상황 | 하이브리드 — 원격 기준으로 설계, 같은 공간은 특수케이스로 흡수 |
| 핵심 단위 | **지속되는 파티 방** (방 안에서 게임을 갈아끼움) |
| UI 기조 | 단순 UI 허용 (디자이너 없음 → 공유 UI키트로 구조적 정당화) |

**"단순 UI"의 명분과 메커니즘:** 디자인을 게임마다 새로 하지 않고, 허브에 **공유 UI 키트를 1회** 정의한 뒤 모든 게임이 조립만 한다. 게임당 디자인 비용 ≈ 0.

## 2. 저작권 정리 (해결됨)

- 게임의 **메커니즘·규칙은 저작권 보호 대상이 아님** (미국·한국 공통).
- 보호 대상: ①이름("The Dalmuti") ②캐릭터/계급 명칭 ③카드 일러스트·아트 ④룰북 문장 표현.
- 안전선: **계급 시스템 제거 + 이름 변경(Upstage) + 자체 아트/네이밍**. → 트레이드오프 없이 안전.
- 허브명도 "달무티"와 무관한 **OneShot**.

---

## 3. 플랫폼 전략

**웹 네이티브 먼저 → 나중에 앱 → 근접감지는 앱 전용 보너스.**

- **프론트엔드:** React + Vite + TypeScript, 상태관리 Zustand.
- **이유:** 파티 도구의 1순위는 "링크/QR 한 번으로 무설치 즉시 참여". Flutter Web은 canvas 렌더링 + 엔진 2MB+ 로 콜드스타트가 느리고 웹 어색함 → 1순위 목표와 충돌. 웹 네이티브가 로딩·웹다움·반복속도 우위.
- **앱화:** 같은 코드를 **Capacitor**로 감싸 iOS/안드 앱 빌드 (추가 코드 거의 없음).
- **근접 자동감지(같은 OS):** 앱 버전 전용 보너스로 후순위. (근거는 §3.2)

### 3.1 조인(참여) 방식 — 3종

1. **링크 공유** — `oneshot.../r/<코드>` (카톡/메시지)
2. **방 코드 입력** — 4~6자리
3. **QR 코드** — 방장 화면 QR → 카메라로 스캔 즉시 입장. **iOS·안드 공통**, 사실상 "옆사람 바로 참여"의 현실적 정답.

### 3.2 근접 자동감지 — 의사결정 근거 (후순위)

- "앱 켜면 수동 조작 없이 주변 같은 OS 사람 자동 감지"는 **같은 OS 한정 구현 가능** (iOS Multipeer / 안드 Nearby). 단 ①최초 1회 권한 팝업 ②해당 무선이 OS에서 켜져 있어야 함.
- **iOS↔안드는 불가** (서로 다른 독립 기술, 호환 안 됨). 혼합 모임에선 코드/QR 폴백 불가피.
- **진짜 마찰은 "근접"이 아니라 "전원 앱 설치"** — 근접 매직은 설치를 없애주지 못하므로 첫 모임엔 무용. 단골 그룹 재회 시에만 가치.
- **결론:** 웹+QR이 그 경험의 90%를 5% 비용에 제공. 근접은 앱 단계의 "있으면 좋은" 보너스로 미룸.

## 4. 백엔드 & 실시간

- **Colyseus (Node + TS) on Oracle Cloud 무료 인스턴스(4코어/24GB).**
- **권위 서버(authoritative):** 카드 배분·턴 검증·역할/정답·패 숨김을 **서버만 보유**. 클라엔 자기 것만 전송 → 치팅·훔쳐보기 불가.
- **멀티방 격리:** Colyseus는 방마다 독립 인스턴스 → A방이 B방에 간섭 불가(구조적 보장). 4코어로 동시 수백 방 수용.
- **데이터 영속성:** 게임 진행 상태는 서버 메모리(Colyseus state). 계정 없음(익명 + 닉네임). 전적 등 최소 영속 데이터가 필요해지면 그때 SQLite/Postgres 추가.

## 5. 아키텍처 — 방(PartyRoom)과 게임(GameModule) 분리

핵심 원칙: **"방"이 영속 단위, "게임"은 그 안에 꽂히는 모듈.** PartyRoom은 게임 규칙을 모른다.

```
client/  (React + Vite + TS, Zustand)
  lobby/         메인: 방 만들기 / 코드·QR로 입장
  room/          파티 방: 멤버 목록, 게임 선택 메뉴, (옵션) 채팅
  ui-kit/        ★ 공유 UI 키트 (버튼·카드·모달·플레이어목록·색토큰)
  games/
    kinggame/    게임별 화면
    upstage/
    liar/ ...

server/  (Colyseus, Node + TS)
  PartyRoom.ts   ★ 영속 Room = 파티 방
                 - 멤버 관리, 무제한 재접속, 방장 권한·임시 이양, 수동 추방
                 - 현재 활성 GameModule 로드/언로드
  games/
    GameModule.ts (interface)   ← 모든 게임이 따르는 계약
    kinggame/KingGame.ts
    upstage/UpstageGame.ts ...
  core/          공통: TurnManager, SecretDealer, PhaseMachine, Voting, Randomizer, ResultRanking
  bots/          QA용 봇 플레이 하니스

shared/  (client·server 공유)
  types, 게임 메타데이터, 룰 상수
```

### 5.1 GameModule 인터페이스 (계약)

새 게임 = 이 인터페이스 구현 하나. 허브 확장 = `games/`에 파일 추가.

- `start(players)` — 시작/초기 상태
- `handleAction(player, action)` — 액션 검증·반영
- `getState(forPlayer)` — 해당 플레이어 관점 상태(패·역할 숨김 처리)
- `onPlayerLeave(player)` / `onPlayerReturn(player)`
- `isOver()` → 결과(ResultRanking: 순위/승자, 없으면 null)

### 5.2 공통 1차 모듈 (5개 게임에서 추출)

| 모듈 | 역할 | 쓰는 게임 |
|---|---|---|
| TurnManager | 턴 순서 관리 | Upstage, 사칙연산 |
| SecretDealer | 비밀 배정·패 숨김 | Upstage, 라이어, 바보라이어, 왕게임 |
| PhaseMachine | 배정→플레이→투표/공개→결과 | 라이어, 바보라이어 |
| Voting | 투표 집계 | 라이어, 바보라이어 |
| Randomizer | 랜덤 역할 배정 | 왕게임 |
| ResultRanking | 결과→다음게임 선택권·전적 | 전체 |

## 6. 방 수명주기 정책 (무기한 대기 원칙)

**대원칙: 시스템은 절대 자동으로 스킵·추방하지 않는다. 사람(방장)만 개입한다.**

- **턴 타이머 없음:** 그 사람 턴이면 **무기한 대기**. 임의 패스 없음.
- **"끊김" 정의를 느슨하게:** 알림바 내리기, 잠깐 다른 앱 보기, 화면 잠깐 끔 → 전부 정상, 자리 유지. 실제 소켓이 떨어져도 "나감"으로 처리하지 않음.
  - 기술 사실: 모바일 OS는 백그라운드 앱의 소켓을 강제로 끊을 수 있음. → "끊김=나감"이 아니라 **세션 토큰**으로 돌아오면 자동 재접속해 **같은 자리에 복원**. Colyseus 재접속을 **무제한**으로 설정.
  - 화면엔 "자리비움/재접속 중" 표시만 띄움. 게임은 그냥 기다림.
- **추방 = 방장(또는 임시 대행) 전용, 수동만.** 시스템 자동 추방 0건. 끊긴 사람이 영영 안 와 판이 멈추면 해결책은 **방장이 직접 추방**(= 사람이 정하는 타임아웃). 방장이 턴 주인을 추방하면 그 순간 방장 행동으로 그 턴 정리.
- **게임 선택 화면에서 끊김도 동일** — 추방 아님, 무기한 대기.
- **빈 방 청소(별개):** 방 안 전원이 장시간 0명 접속이면 그 빈 방 자체만 메모리에서 정리. 한 명이라도 붙어있으면 안 건드림.

### 6.1 호스트(방장) 모델 — 고정 방장

- **고정 방장:** 방 만든 사람이 계속 방장. 추방·방 설정·다음 게임 선택권 보유.
- **다음 게임 선택권:** 기본 방장이 쥐되, 원하면 투표로 개방 가능.
- **임시 이양:** 방장이 끊기면 **가장 오래된 접속 멤버**가 임시 대행. 원 방장 복귀 시 자동 반환.

## 7. 게임 카탈로그

| 게임 | 핵심 메커닉 | 무게 |
|---|---|---|
| **Upstage** | 달무티 기반 턴제 카드, 패 숨김, 같은 수량 더 낮은 숫자로 이기기 | ★★★ |
| **라이어** | 역할/단어 비밀배정 → (대화) → 투표 → 공개 | ★ |
| **바보라이어** | 라이어의 변형(config) — 코드 대부분 공유 | ★ |
| **사칙연산** | 빠른 턴제 연산 체인, 틀리면 패배 | ★ |
| **왕게임** | 랜덤 역할 배정 + 왕 공개 | ☆ |

(추후 게임 계속 추가 예정. 추가 = GameModule 구현 + 화면 조립.)

## 8. 제작 파이프라인 (게임 1개 추가 = 반복 템플릿)

가로지르는 인프라(방·조인·재접속·호스트·UI키트·모듈로더·봇하니스)는 **허브-라이트 단계에 1회** 구축. 이후 게임마다 5스텝 반복:

1. **기획** — 1페이지 게임 스펙(규칙·페이즈·인원·액션·승리조건·숨김정보·엣지) → `docs/games/<game>.md`. (유준상 주도, Claude 초안)
2. **디자인** — 공유 UI키트 재사용. 게임 고유 화면만 조립.
3. **개발** — `GameModule`(서버 순수 규칙, TDD) + 게임 화면(클라).
4. **QA** — 규칙 유닛테스트 + 봇 N명 자동 플레이 하니스 + 실제 파티 플레이테스트 + 재접속/추방 체크리스트.
5. **통합** — 허브 카탈로그 등록 → 실제 방 스모크 테스트.

**역할 배치:** 유준상 = 총괄+기획 / Claude = 개발+공유디자인+QA자동화.

## 9. 로드맵

- **Phase 0 — 허브-라이트 (걷는 해골):** PartyRoom + 로비 + 조인 3종(링크/코드/QR) + 무제한 재접속 + 호스트/추방 + 공유 UI키트 + GameModule 인터페이스·로더 + Oracle 배포 + 봇 하니스.
- **Phase 1 — 왕게임:** 가장 단순(랜덤배정+공개). 방→게임→결과→방복귀 전체 루프·인프라를 최소 복잡도로 검증.
- **Phase 2 — Upstage:** 가장 어려운 케이스(패 숨김·턴 검증·턴 중간 재접속)를 안정된 토대 위에서.
- **Phase 3 — 라이어 / 바보라이어:** PhaseMachine·Voting·SecretDealer 재사용. 두 게임이 코드 공유.
- **Phase 4 — 사칙연산:** TurnManager 재사용.
- **이후:** 게임 계속 추가.

## 10. 비범위 (YAGNI — 지금 하지 않음)

- 계정/로그인 시스템 (익명+닉네임으로 충분)
- 수익화(광고·IAP·코인 배팅) — 예전 Upstage 기획의 사행성 등급 이슈 회피
- 코드 레벨 허브 추상화의 과도한 사전 일반화 — 2번째 게임(Upstage) 만들 때 실제 경계로 추출
- 근접 자동감지 — 앱 단계의 보너스로 미룸
- DB 영속화 — 전적 기능이 실제로 필요해질 때
